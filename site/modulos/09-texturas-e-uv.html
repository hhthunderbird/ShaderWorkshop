Texturas & UV

Seu cubo do M7 era colorido por contas. Mas e quando você quer vestir o objeto com uma imagem de verdade — madeira, tijolo, o rosto de um personagem? Isso é uma textura. E o truque que cola a imagem na superfície tem nome que você já conhece de outro lugar: UV.

Textura = uma imagem que o shader consulta

Uma textura é só uma imagem. O shader, pra cada pixel da superfície, pergunta: "qual cor da imagem eu uso aqui?". A função que responde é texture2D(imagem, uv): você dá um endereço uv (de 0 a 1) e ela devolve a cor daquele ponto da imagem.

Você já viu "UV" antes, lá no M2 — mas cuidado: o sentido mudou. No M2, v_uv era a posição do pixel na tela. Aqui, UV é o endereço na imagem/superfície: que pedacinho da figura cola em cada ponto do objeto. Mesma faixa 0 a 1, significado novo.

Vestindo o objeto 3D

O mesmo texture2D(u_tex, v_uv), agora num cubo girando. Cada face do cubo tem suas coordenadas UV, então a imagem cola direitinho em cada lado. Repare na barra branca e no pontinho preto pra conferir que a imagem não está de cabeça pra baixo.

🧮 Matemática de bolso: por que fract repete (recordando o M3)

fract(x) guarda só a parte depois da vírgula: fract(2.3) = 0.3. Então, conforme v_uv vai de 0 a 1, v_uv * 3.0 vai de 0 a 3, e o fract faz isso "dar a volta" três vezes: 0→1, 0→1, 0→1. Cada volta é um ladrilho. Trocar o 3.0 muda quantos ladrilhos.

🗣️ Dois sotaques: texture2D no Unity

A ideia é a mesma; no Unity moderno (HLSL) a amostragem se escreve diferente (tex.Sample(sampler, uv)), mas o conceito — endereço UV de 0 a 1 → cor da imagem — é idêntico. Clique 🔁 Ver em HLSL pra uma ideia (é ilustrativo).

A imagem precisa ter algum tamanho especial?
No WebGL 1 (a versão que roda no navegador), imagens com lados potência de 2 (tipo 128×128) funcionam com todas as configurações de repetição e mipmap. Tamanhos fora disso (NPOT) têm restrições da API — não é o hardware, é o WebGL 1. O motor já usa imagens comportadas pra você não precisar se preocupar.
Por que minha textura aparece de cabeça pra baixo às vezes?
Porque o "0,0" da imagem e o "0,0" da UV podem ficar em cantos diferentes. O motor já vira a imagem pra ela aparecer em pé — mas é o tropeço clássico de textura. Use a barra branca do topo pra checar.
Dá pra usar várias imagens?
Dá (cada uma num "sampler"), mas aqui a gente fica com uma só. Múltiplas texturas é assunto de cursos mais longos.
UV mudou de sentido. No M2 a UV era a posição na tela; aqui é o endereço na imagem/superfície. Se você pensar "tela" no lugar de "superfície", vai colar a imagem errado. Mesma faixa 0..1, ideia diferente.
  • Textura = imagem que o shader consulta com texture2D(u_tex, uv).
  • UV = endereço de 0 a 1 na imagem/superfície (não é a tela, como era no M2).
  • fract(uv * N) faz a imagem repetir em N ladrilhos (recupera o M3).
  • Cada face/ponto do objeto tem sua UV — por isso a imagem cola certo.
